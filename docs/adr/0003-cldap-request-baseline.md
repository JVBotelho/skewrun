# ADR-0003: CLDAP Request Baseline — rootDSE Query, Not DC Locator Ping

**Status:** Accepted  
**Date:** 2026-06-21

## Context

The original code and README described the CLDAP method as emulating a
"Windows DC Locator Ping (`rootDSE`)." This was factually incorrect on two counts.

**What a DC Locator Ping actually is:**

A Windows DC Locator Ping (MS-ADTS §7.3.2) is a CLDAP Search Request with:
- Filter: `(&(DnsDomain=X)(NtVer=0x00000006)(AAC=0x00000000)(DnsHostName=Y)...)`
- Attribute: `Netlogon`

This is machine-account traffic generated by every Windows workstation multiple times per
hour to locate and validate domain controllers.

**What the code actually sends:**

```
Base DN:  ""  (rootDSE)
Scope:    baseObject
Filter:   (objectClass=*)
Attrs:    [schemaNamingContext, namingContexts, currentTime, dnsHostName, supportedLDAPVersion]
```

This is a standard LDAP rootDSE diagnostic query — the pattern of `ldapsearch -s base`,
`PowerShell Get-ADRootDSE`, LDP.exe, Softerra LDAP Browser, and monitoring agents
(SolarWinds SAM, Nagios LDAP check, Zabbix AD plugin). It is **not** DC Locator traffic.

The distinction matters operationally. An operator who believes the traffic hides in DC
Locator volume (hundreds per hour per workstation) will miscalibrate their detection
expectations. The actual hiding pool is admin tool / monitoring traffic — lower absolute
volume but with a harder-to-fingerprint packet structure.

**`timeLimit` field:**

The initial implementation randomized `timeLimit` in `10..=30` seconds, with a comment
claiming this looks like a normal client. No documented Windows tool or standard LDAP client
library sends `timeLimit` in the 10-30 range for rootDSE queries.

RFC 4511 §4.5.1 defines `timeLimit = 0` as "no client-requested time limit restrictions."
This is the value sent by `ldapsearch` (without `-l`), `PowerShell Get-ADRootDSE`, and
monitoring agents. A randomized non-zero value in a range with no documented baseline is
self-generated noise that narrows rather than widens the tool's hiding pool.

## Decision

1. Correct all documentation (README table, module-level doc comment) to accurately describe
   the traffic as a rootDSE diagnostic query matching the `ldapsearch`/PowerShell baseline —
   not a DC Locator Ping.

2. Set `timeLimit = 0` in the LDAP Search Request, consistent with RFC 4511 and the observed
   behavior of the tools this query is designed to blend into.

The `messageID` randomization (1..=1000) and the attribute list dilution are retained: they
serve a legitimate purpose (breaking static per-packet signatures and making `currentTime`
appear as a secondary interest alongside standard rootDSE attrs).

## Consequences

- OPSEC model is now accurately documented: the tool hides in admin LDAP tool traffic, not
  DC Locator ping traffic.
- `timeLimit = 0` removes a field that was previously a small but unnecessary deviation from
  the declared baseline.
- The tool cannot be made to look like a DC Locator Ping without a full filter and attribute
  rewrite (different scope, filter, attribute set, and response parsing). This is out of
  scope — the rootDSE baseline is legitimate, well-documented, and sufficient for the tool's
  purpose.
